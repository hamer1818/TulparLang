# Benchmarks in CI

The numbers shown in [`README.md`](../README.md) and
[`RESULTS.md`](RESULTS.md) are regenerated by GitHub Actions on every
push to `main`. This document covers the runner identity, what we
actually measure, and how to read the numbers — particularly when CI
variance kicks in.

## Why this is automated

Hand-edited benchmark numbers drift. Someone runs the suite locally,
copies the output into the README, ships the PR — and a year later the
prose still says "1.7× gcc" even after the AOT pipeline has changed
five times. The numbers stop being a measurement and start being
folklore.

CI fixes that. Every push to `main` rebuilds Tulpar, rebuilds every
comparator (C, Rust, Go, Node, Python), runs the same workloads, and
rewrites the marker-managed sections of the README and RESULTS.md from
fresh data. The numbers can never be older than the last commit on
`main`. We don't get to claim a result that the runner can't reproduce.

## Workflow

[`.github/workflows/bench.yml`](../.github/workflows/bench.yml) runs:

1. Checks out the commit.
2. Installs build deps (LLVM 18, Go, Node, Python, gcc, ccache; Rust
   is preinstalled on `ubuntu-latest`).
3. Builds Tulpar (`Release`, ccache-accelerated).
4. Runs [`benchmarks/ci_run.py`](ci_run.py), which builds every
   comparator binary and runs each workload 5 times, taking the best
   wall time. Writes `benchmarks/RESULTS.json` (machine-readable) and
   regenerates the auto-managed Markdown block in `RESULTS.md`.
5. Runs [`benchmarks/update_readme.py`](update_readme.py), which
   replaces the content between the `<!-- BENCH:* START/END -->`
   markers in `README.md` from the same JSON.
6. Commits and pushes the refreshed files back to `main` if anything
   changed. The commit message contains `[bench skip]` so the bot's
   own commit can't trigger another bench run; pushes performed by the
   workflow's `GITHUB_TOKEN` also don't fire `push` workflows by
   default, so this is belt-and-suspenders.

## Runner identity

GitHub-hosted `ubuntu-latest` runners. As of writing that's a 2-core /
~7 GB-RAM VM on shared infrastructure. The exact CPU model and kernel
are captured in `RESULTS.json` per run (`run.runner.cpu_model`,
`run.runner.os`, `run.runner.release`) and shown at the top of the
auto-managed RESULTS.md block.

We do **not** use self-hosted or larger runners. The baseline is
deliberately commodity hardware so the comparison stays apples-to-
apples across PRs and so anyone can reproduce on a similar laptop.

## What we measure

**CPU benchmarks.** Two workloads:

- `loopsum`: `while (i < 10_000_000) { total += i; i++; }`. Stresses
  loop codegen and integer ALU.
- `fib(35)`: classic recursive Fibonacci. Stresses function-call
  overhead, branch prediction, register allocation around recursion.

Both read their iteration count / depth from `TULPAR_BENCH_N` so LLVM
at `-O3` can't fold them to closed-form. Each language runs the same
workload, compiled with its idiomatic optimization flags
(`gcc -O2`, `rustc -C opt-level=3`, `go build` defaults). Best wall
time of 5 runs.

**HTTP throughput.** 3000 GETs across 4 keep-alive connections,
single-machine localhost. Measured against the four Wings listener
variants (`listen`, `listen_async`, `listen_pool x8`, `listen_evented`)
and two reference servers (Node.js' built-in `http`, Python's
`ThreadingHTTPServer`). All servers return the same JSON body.
`TULPAR_HTTP_QUIET=1` is set so per-request access logs don't bias
toward Tulpar's I/O stack.

## What we don't measure

- **Production workloads.** No real backpressure, no slow clients, no
  large payloads, no HTTPS handshake cost. Localhost loopback skips
  the whole network stack. Don't read these as production-traffic
  projections.
- **p99 / tail latency.** We report mean throughput. A server can win
  RPS while losing on tail latency; that distinction needs a different
  benchmark.
- **Cold start.** Tulpar AOT binaries amortize startup amortizes over
  3000 requests; for sub-100-request workloads results differ.
- **Java.** JVM startup dominates the short timings and varies
  enormously on shared CI runners — we'd measure runner-scheduler
  noise more than the JIT. The local `run_benchmarks.sh` still runs
  Java for completeness; the CI workflow passes `--no-java`.

## Reading the numbers (CI variance)

Shared CI runners produce **noisy absolute timings**. Run-to-run
variance on `ubuntu-latest` for a 30 ms benchmark is routinely 10-30 %
because the underlying VM shares CPU and memory bandwidth with other
tenants. This is a fundamental property of GitHub-hosted runners, not
something the benchmark harness can paper over.

What the harness already does:

- **Best-of-5.** We take the minimum wall time across 5 runs. This
  filters out runs that lost the CPU to a co-tenant mid-execution.
- **Same runner, same job.** Every comparator runs in the same job on
  the same VM in close temporal succession, so they all observe the
  same noise. Ratios (Tulpar/gcc, Tulpar/Node) are far more stable
  across CI runs than absolute milliseconds.

**How to read a result, in priority order:**

1. **Ratios first.** "Tulpar AOT is 1.5× gcc -O2 on loopsum" is
   meaningful across CI runs. The absolute "31.4 ms" varies more.
2. **Trends across multiple runs.** A single CI run that shows fib at
   45 ms instead of 55 ms isn't a 20% AOT improvement — it's
   probably a quieter runner that day. Watch the metric over a
   handful of consecutive `main` commits.
3. **Absolute numbers** are last. They're correct for the run that
   produced them, but they don't transfer to your laptop or to a
   production server.

If you need single-machine, controlled-environment numbers — the kind
you'd cite in a paper — run `benchmarks/ci_run.py` locally on a quiet
machine with no other workloads. The script is the same one CI runs;
the variance just disappears when you control the host.

## Local reproduction

```bash
# Build Tulpar first.
./build.sh

# Run the same orchestrator the CI uses.
python3 benchmarks/ci_run.py --repeats 5

# (Optional) preview how update_readme.py would rewrite README.md
# without committing.
python3 benchmarks/update_readme.py
```

Outputs:

- `benchmarks/RESULTS.json` — machine-readable, full result set
  including runner metadata.
- `benchmarks/RESULTS.md` — auto-managed Markdown block updated;
  prose context preserved.
- `README.md` — performance tables and meta-block updated.

## When this breaks

- **Toolchain missing on the runner.** `ci_run.py` skips a missing
  language and emits a `—` cell rather than failing. If that happens
  in CI it usually means the GHA runner image dropped a preinstalled
  toolchain — check the workflow's "Run benchmarks" step log and add
  an explicit `setup-*` action.
- **Tulpar AOT compile fails.** The orchestrator exits non-zero,
  which fails the workflow. The bench commit is skipped — README/
  RESULTS.md keep the previous numbers. Fix the AOT regression on
  `main` and the next push will refresh.
- **Marker drift in README.** If someone removes a `<!-- BENCH:* -->`
  marker pair, `update_readme.py` skips that section silently. The
  CPU/HTTP tables would freeze at their last value while CI keeps
  refreshing `RESULTS.json`. Restore the markers and the next run
  re-syncs.
- **Branch protection on main.** If `main` becomes protected and the
  workflow's `GITHUB_TOKEN` loses push permission, the bot commit
  step will fail. Either (a) allow `tulpar-bench-bot` to bypass
  protection, or (b) switch the push to a PR-creation step instead
  of a direct push. The current setup assumes the personal-repo
  default of unprotected `main`.
